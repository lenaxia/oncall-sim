## 1. Session Model (`session/session.ts`)

```typescript
export interface Session {
  id:          string
  scenarioId:  string
  scenario:    LoadedScenario
  gameLoop:    GameLoop
  // metrics are owned by the game loop (passed in via GameLoopDependencies)
  // and returned via gameLoop.getSnapshot().metrics — not stored separately here
  debrief:     DebriefResult | null   // null until resolve is called and LLM completes
  createdAt:   number   // real wall-clock ms
  lastSseAt:   number   // real wall-clock ms of last SSE connection
  status:      SessionStatus
}

export type SessionStatus = 'active' | 'resolved' | 'expired'

export interface DebriefResult {
  narrative:         string             // generated by Debrief LLM in Phase 9; empty string until then
  evaluationState:   EvaluationState   // from Phase 4 evaluator
  auditLog:          AuditEntry[]
  eventLog:          SimEventLogEntry[] // simulation events for debrief timeline (excludes sim_time/session_snapshot)
  resolvedAtSimTime: number
}
```

## 2. Session Store (`session/session-store.ts`)

```typescript
export interface SessionStore {
  create(session: Session): void
  get(sessionId: string): Session | null
  delete(sessionId: string): void
  getAll(): Session[]

  // Evicts sessions with no SSE connection for > SESSION_EXPIRY_MS (10 minutes)
  evictExpired(): void
}

export function createSessionStore(): SessionStore
```

The session store has no persistence — all state is in-memory. A server restart loses all sessions. Session expiry runs on a background interval (every 60 real seconds).

## 3. Session Factory

The session factory is the only place where all Phase 1–5 components are assembled and wired together. Lives in `session/session.ts`.

```typescript
export async function createSession(
  scenarioId: string,
  scenario:   LoadedScenario,
  llmClient:  LLMClient
): Promise<Session> {
  const sessionId = generateSessionId()  // uuid v4

  // Generate metrics for this session's unique noise profile
  const metrics = generateAllMetrics(scenario, sessionId)

  // Build engine components
  const clock     = createSimClock(scenario.timeline.defaultSpeed)
  const scheduler = createEventScheduler(scenario)
  const auditLog  = createAuditLog()
  const store     = createConversationStore()
  const evaluator = createEvaluator()

  // Pre-populate conversation store with scenario's initial state
  // (tickets, deployments, wiki, cicd from scenario config)
  populateInitialState(store, scenario)

  // Build stakeholder engine
  const stakeholderEngine = createStakeholderEngine(llmClient, scenario)

  // Build game loop with stakeholder hook wired in
  const gameLoop = createGameLoop({
    scenario,
    sessionId,
    clock,
    scheduler,
    auditLog,
    store,
    evaluator,
    metrics,
    onDirtyTick: (ctx) => stakeholderEngine.tick(ctx),
  })

  return {
    id:         sessionId,
    scenarioId,
    scenario,
    gameLoop,
    debrief:    null,
    createdAt:  Date.now(),
    lastSseAt:  Date.now(),
    status:     'active',
  }
}
```

`populateInitialState` seeds the conversation store with the scenario's pre-configured tickets, CI/CD deployments, and any scripted events at `t < 0` (e.g. a deployment that happened before the incident).

## 4. SSE Broker (`sse/sse-broker.ts`)

Manages SSE connections per session. Multiple connections for the same session are allowed (tab refresh before old connection closes).

```typescript
export interface SSEBroker {
  // Register a new SSE connection for a session.
  // Immediately sends session_snapshot as the first event.
  // Returns a cleanup function — call on connection close.
  connect(sessionId: string, res: Response): () => void

  // Broadcast a SimEvent to all active connections for a session.
  broadcast(sessionId: string, event: SimEvent): void

  // Returns the number of active connections for a session (for health checks).
  connectionCount(sessionId: string): number
}

export function createSSEBroker(sessionStore: SessionStore): SSEBroker
```

SSE wire format:
```
data: {"type":"session_snapshot","snapshot":{...}}\n\n
data: {"type":"chat_message","channel":"#incidents","message":{...}}\n\n
```

Each connection has a heartbeat (comment line `: heartbeat\n\n`) every 15 real seconds to prevent proxy timeouts.

On connect:
1. Set response headers (`Content-Type: text/event-stream`, `Cache-Control: no-cache`, `Connection: keep-alive`)
2. Get session from session store — if not found, send `session_expired` event
3. Update `session.lastSseAt = Date.now()`
4. Send `session_snapshot` immediately (full state from `gameLoop.getSnapshot()`)
5. Register `gameLoop.onEvent` handler to forward events to this connection
6. Return cleanup function that removes the handler on connection close

## 6. Error Handling

All routes follow a consistent pattern:

```typescript
// 400: malformed request body (Zod validation on request body)
// 404: session or scenario not found
// 409: action not valid for current session status (e.g. resolve on already-resolved session)
// 500: unexpected server error — logged, generic message returned to client
```

No route throws. All async errors are caught and passed to Express error handler.

```typescript
// Global error handler in index.ts
app.use((err: Error, req: Request, res: Response, next: NextFunction) => {
  logger.error({ err, path: req.path }, 'Unhandled route error')
  res.status(500).json({ error: 'Internal server error' })
})
```

## 7. App Entry Point (`index.ts`)

```typescript
async function main() {
  // 1. Validate env config — exit with clear message if required vars missing
  const config = loadConfig()

  // 2. Load all scenarios
  const scenarios = await loadAllScenarios(config.scenariosDir)
  logger.info({ count: scenarios.size }, 'Scenarios loaded')

  // 3. Create LLM client
  const llmClient = createLLMClient()

  // 4. Create session store + SSE broker
  const sessionStore = createSessionStore()
  const sseBroker    = createSSEBroker(sessionStore)

  // 5. Start session expiry loop (every 60 real seconds)
  setInterval(() => sessionStore.evictExpired(), 60_000)

  // 6. Mount routes
  const app = express()
  app.use(express.json())
  app.use('/api/scenarios', scenariosRouter(scenarios))
  app.use('/api/sessions',  sessionsRouter(sessionStore, sseBroker, scenarios, llmClient))
  app.use(errorHandler)

  // 7. Start server
  app.listen(config.port, () => {
    logger.info({ port: config.port }, 'Server started')
  })
}
```

## 9. Reconnection and Session Expiry

From HLD §19:

- Sessions survive SSE disconnection — game loop keeps running
- On SSE reconnect: `session_snapshot` sent immediately as first event
- Sessions with no SSE connection for > `SESSION_EXPIRY_MS` are evicted by `sessionStore.evictExpired()`
- Client reconnects after expiry: `session_expired` event sent, client redirects to picker

The SSE broker updates `session.lastSseAt` on each new connection. The expiry check uses this timestamp.

### `session-store.test.ts` (unit)

```
createSessionStore:
  - create and get returns the session
  - get unknown id returns null
  - delete removes session
  - evictExpired removes sessions with lastSseAt > expiry threshold
  - evictExpired does not remove recently-connected sessions
```

### `sse-broker.test.ts` (unit)

```
connect:
  - first event sent is session_snapshot
  - session_snapshot contains correct sessionId and simTime
  - connect to expired/unknown session sends session_expired event
  - cleanup function called on disconnect removes event handler

broadcast:
  - event delivered to all active connections for session
  - event not delivered to connections for other sessions
```

### Integration tests (`routes/*.test.ts`)

All integration tests:
- Start real Express app with `MOCK_LLM=true`
- Use fixture scenario
- Use `supertest` for HTTP requests
- Verify SSE events with a simple EventSource mock

```

  - valid scenarioId → 201 with sessionId
  - unknown scenarioId → 404
  - game loop started after session creation

  - scripted event at t=0 delivered after session starts
  - session_expired sent on reconnect after expiry

  - valid action → 204
  - invalid action type → 400
  - unknown session → 404

  - chat_message SSE event broadcast with trainee's message

  - email_received SSE event broadcast with reply

  - 204, session removed, game loop stopped
```
